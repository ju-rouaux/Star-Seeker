If you are looking to queue audio for output on a non-\/callback playback device, you want \mbox{\hyperlink{SDL__audio_8h_a19ef6fa2396f581aaf4d9a9512972268}{SDL\+\_\+\+Queue\+Audio()}} instead. \mbox{\hyperlink{SDL__audio_8h_a6ed6333b8b294eafb6a9435fd8ff89fa}{SDL\+\_\+\+Dequeue\+Audio()}} will always return 0 if you use it with playback devices.

SDL offers two ways to retrieve audio from a capture device\+: you can either supply a callback that SDL triggers with some frequency as the device records more audio data, (push method), or you can supply no callback, and then SDL will expect you to retrieve data at regular intervals (pull method) with this function.

There are no limits on the amount of data you can queue, short of exhaustion of address space. Data from the device will keep queuing as necessary without further intervention from you. This means you will eventually run out of memory if you aren\textquotesingle{}t routinely dequeueing data.

You may not dequeue audio from a device that is using an application-\/supplied callback; doing so returns an error. You have to use the audio callback, or dequeue audio with this function, but not both.

{\footnotesize\ttfamily DECLSPEC \mbox{\hyperlink{structSDL__AudioSpec}{SDL\+\_\+\+Audio\+Spec}}$\ast$ SDLCALL SDL\+\_\+\+Load\+WAV\+\_\+\+RW (\begin{DoxyParamCaption}\item[{\mbox{\hyperlink{structSDL__RWops}{SDL\+\_\+\+RWops}} $\ast$}]{src,  }\item[{int}]{freesrc,  }\item[{\mbox{\hyperlink{structSDL__AudioSpec}{SDL\+\_\+\+Audio\+Spec}} $\ast$}]{spec,  }\item[{Uint8 $\ast$$\ast$}]{audio\+\_\+buf,  }\item[{Uint32 $\ast$}]{audio\+\_\+len }\end{DoxyParamCaption})}

Load the audio data of a WAVE file into memory.

Loading a WAVE file requires {\ttfamily src}, {\ttfamily spec}, {\ttfamily audio\+\_\+buf} and {\ttfamily audio\+\_\+len} to be valid pointers. The entire data portion of the file is then loaded into memory and decoded if necessary.

If {\ttfamily freesrc} is non-\/zero, the data source gets automatically closed and freed before the function returns.

Supported formats are RIFF WAVE files with the formats PCM (8, 16, 24, and 32 bits), IEEE Float (32 bits), Microsoft ADPCM and IMA ADPCM (4 bits), and A-\/law and mu-\/law (8 bits). Other formats are currently unsupported and cause an error.

If this function succeeds, the pointer returned by it is equal to {\ttfamily spec} and the pointer to the audio data allocated by the function is written to {\ttfamily audio\+\_\+buf} and its length in bytes to {\ttfamily audio\+\_\+len}. The \mbox{\hyperlink{structSDL__AudioSpec}{SDL\+\_\+\+Audio\+Spec}} members {\ttfamily freq}, {\ttfamily channels}, and {\ttfamily format} are set to the values of the audio data in the buffer. The {\ttfamily samples} member is set to a sane default and all others are set to zero.

It\textquotesingle{}s necessary to use \mbox{\hyperlink{SDL__audio_8h_a631aa8624575603df2c5ab9257feee59}{SDL\+\_\+\+Free\+WAV()}} to free the audio data returned in {\ttfamily audio\+\_\+buf} when it is no longer used.

Because of the underspecification of the .WAV format, there are many problematic files in the wild that cause issues with strict decoders. To provide compatibility with these files, this decoder is lenient in regards to the truncation of the file, the fact chunk, and the size of the RIFF chunk. The hints {\ttfamily SDL\+\_\+\+HINT\+\_\+\+WAVE\+\_\+\+RIFF\+\_\+\+CHUNK\+\_\+\+SIZE}, {\ttfamily SDL\+\_\+\+HINT\+\_\+\+WAVE\+\_\+\+TRUNCATION}, and {\ttfamily SDL\+\_\+\+HINT\+\_\+\+WAVE\+\_\+\+FACT\+\_\+\+CHUNK} can be used to tune the behavior of the loading process.

Use this function to lock out the audio callback function for a specified device.

The lock manipulated by these functions protects the audio callback function specified in \mbox{\hyperlink{SDL__audio_8h_af54523e415de7c7fdd0d48a76ab58bb3}{SDL\+\_\+\+Open\+Audio\+Device()}}. During a \mbox{\hyperlink{SDL__audio_8h_a9769845303f88f084f051b99e55b62a1}{SDL\+\_\+\+Lock\+Audio\+Device()}}/\+SDL\+\_\+\+Unlock\+Audio\+Device() pair, you can be guaranteed that the callback function for that device is not running, even if the device is not paused. While a device is locked, any other unpaused, unlocked devices may still run their callbacks.

Calling this function from inside your audio callback is unnecessary. SDL obtains this lock before calling your function, and releases it when the function returns.

You should not hold the lock longer than absolutely necessary. If you hold it too long, you\textquotesingle{}ll experience dropouts in your audio playback. Ideally, your application locks the device, sets a few variables and unlocks again. Do not do heavy work while holding the lock for a device.

It is safe to lock the audio device multiple times, as long as you unlock it an equivalent number of times. The callback will not run until the device has been unlocked completely in this way. If your application fails to unlock the device appropriately, your callback will never run, you might hear repeating bursts of audio, and \mbox{\hyperlink{SDL__audio_8h_abd910f31079d53b6c0ea8bf32dac0ce6}{SDL\+\_\+\+Close\+Audio\+Device()}} will probably deadlock.

Internally, the audio device lock is a mutex; if you lock from two threads at once, not only will you block the audio callback, you\textquotesingle{}ll block the other thread.

Managing the audio spec via \textquotesingle{}desired\textquotesingle{} and \textquotesingle{}obtained\textquotesingle{}\+:

When filling in the desired audio spec structure\+:


\begin{DoxyItemize}
\item {\ttfamily desired-\/$>$freq} should be the frequency in sample-\/frames-\/per-\/second (Hz).
\item {\ttfamily desired-\/$>$format} should be the audio format ({\ttfamily AUDIO\+\_\+\+S16\+SYS}, etc).
\item {\ttfamily desired-\/$>$samples} is the desired size of the audio buffer, in {\itshape sample frames} (with stereo output, two samples--left and right--would make a single sample frame). This number should be a power of two, and may be adjusted by the audio driver to a value more suitable for the hardware. Good values seem to range between 512 and 8096 inclusive, depending on the application and CPU speed. Smaller values reduce latency, but can lead to underflow if the application is doing heavy processing and cannot fill the audio buffer in time. Note that the number of sample frames is directly related to time by the following formula\+: {\ttfamily ms = (sampleframes$\ast$1000)/freq}
\item {\ttfamily desired-\/$>$size} is the size in {\itshape bytes} of the audio buffer, and is calculated by \mbox{\hyperlink{SDL__audio_8h_af54523e415de7c7fdd0d48a76ab58bb3}{SDL\+\_\+\+Open\+Audio\+Device()}}. You don\textquotesingle{}t initialize this.
\item {\ttfamily desired-\/$>$silence} is the value used to set the buffer to silence, and is calculated by \mbox{\hyperlink{SDL__audio_8h_af54523e415de7c7fdd0d48a76ab58bb3}{SDL\+\_\+\+Open\+Audio\+Device()}}. You don\textquotesingle{}t initialize this.
\item {\ttfamily desired-\/$>$callback} should be set to a function that will be called when the audio device is ready for more data. It is passed a pointer to the audio buffer, and the length in bytes of the audio buffer. This function usually runs in a separate thread, and so you should protect data structures that it accesses by calling \mbox{\hyperlink{SDL__audio_8h_a9769845303f88f084f051b99e55b62a1}{SDL\+\_\+\+Lock\+Audio\+Device()}} and \mbox{\hyperlink{SDL__audio_8h_a37427a5d47e10103f2fff8165d5f9000}{SDL\+\_\+\+Unlock\+Audio\+Device()}} in your code. Alternately, you may pass a NULL pointer here, and call \mbox{\hyperlink{SDL__audio_8h_a19ef6fa2396f581aaf4d9a9512972268}{SDL\+\_\+\+Queue\+Audio()}} with some frequency, to queue more audio samples to be played (or for capture devices, call \mbox{\hyperlink{SDL__audio_8h_a6ed6333b8b294eafb6a9435fd8ff89fa}{SDL\+\_\+\+Dequeue\+Audio()}} with some frequency, to obtain audio samples).
\item {\ttfamily desired-\/$>$userdata} is passed as the first parameter to your callback function. If you passed a NULL callback, this value is ignored.
\end{DoxyItemize}

{\ttfamily allowed\+\_\+changes} can have the following flags OR\textquotesingle{}d together\+:


\begin{DoxyItemize}
\item {\ttfamily SDL\+\_\+\+AUDIO\+\_\+\+ALLOW\+\_\+\+FREQUENCY\+\_\+\+CHANGE}
\item {\ttfamily SDL\+\_\+\+AUDIO\+\_\+\+ALLOW\+\_\+\+FORMAT\+\_\+\+CHANGE}
\item {\ttfamily SDL\+\_\+\+AUDIO\+\_\+\+ALLOW\+\_\+\+CHANNELS\+\_\+\+CHANGE}
\item {\ttfamily SDL\+\_\+\+AUDIO\+\_\+\+ALLOW\+\_\+\+ANY\+\_\+\+CHANGE}
\end{DoxyItemize}

These flags specify how SDL should behave when a device cannot offer a specific feature. If the application requests a feature that the hardware doesn\textquotesingle{}t offer, SDL will always try to get the closest equivalent.

For example, if you ask for float32 audio format, but the sound card only supports int16, SDL will set the hardware to int16. If you had set SDL\+\_\+\+AUDIO\+\_\+\+ALLOW\+\_\+\+FORMAT\+\_\+\+CHANGE, SDL will change the format in the {\ttfamily obtained} structure. If that flag was {\itshape not} set, SDL will prepare to convert your callback\textquotesingle{}s float32 audio to int16 before feeding it to the hardware and will keep the originally requested format in the {\ttfamily obtained} structure.

The resulting audio specs, varying depending on hardware and on what changes were allowed, will then be written back to {\ttfamily obtained}.

If your application can only handle one specific data format, pass a zero for {\ttfamily allowed\+\_\+changes} and let SDL transparently handle any differences.

{\footnotesize\ttfamily DECLSPEC void SDLCALL SDL\+\_\+\+Pause\+Audio\+Device (\begin{DoxyParamCaption}\item[{\mbox{\hyperlink{SDL__audio_8h_a677d9d2a6f77a7502a0dc8b45ad211bb}{SDL\+\_\+\+Audio\+Device\+ID}}}]{dev,  }\item[{int}]{pause\+\_\+on }\end{DoxyParamCaption})}

Use this function to pause and unpause audio playback on a specified device.

This function pauses and unpauses the audio callback processing for a given device. Newly-\/opened audio devices start in the paused state, so you must call this function with {\bfseries{pause\+\_\+on}}=0 after opening the specified audio device to start playing sound. This allows you to safely initialize data for your callback function after opening the audio device. Silence will be written to the audio device while paused, and the audio callback is guaranteed to not be called. Pausing one device does not prevent other unpaused devices from running their callbacks.

Pausing state does not stack; even if you pause a device several times, a single unpause will start the device playing again, and vice versa. This is different from how \mbox{\hyperlink{SDL__audio_8h_a9769845303f88f084f051b99e55b62a1}{SDL\+\_\+\+Lock\+Audio\+Device()}} works.

If you just need to protect a few variables from race conditions vs your callback, you shouldn\textquotesingle{}t pause the audio device, as it will lead to dropouts in the audio playback. Instead, you should use \mbox{\hyperlink{SDL__audio_8h_a9769845303f88f084f051b99e55b62a1}{SDL\+\_\+\+Lock\+Audio\+Device()}}.

{\footnotesize\ttfamily DECLSPEC int SDLCALL SDL\+\_\+\+Queue\+Audio (\begin{DoxyParamCaption}\item[{\mbox{\hyperlink{SDL__audio_8h_a677d9d2a6f77a7502a0dc8b45ad211bb}{SDL\+\_\+\+Audio\+Device\+ID}}}]{dev,  }\item[{const void $\ast$}]{data,  }\item[{Uint32}]{len }\end{DoxyParamCaption})}

Queue more audio on non-\/callback devices.

If you are looking to retrieve queued audio from a non-\/callback capture device, you want \mbox{\hyperlink{SDL__audio_8h_a6ed6333b8b294eafb6a9435fd8ff89fa}{SDL\+\_\+\+Dequeue\+Audio()}} instead. \mbox{\hyperlink{SDL__audio_8h_a19ef6fa2396f581aaf4d9a9512972268}{SDL\+\_\+\+Queue\+Audio()}} will return -\/1 to signify an error if you use it with capture devices.

SDL offers two ways to feed audio to the device\+: you can either supply a callback that SDL triggers with some frequency to obtain more audio (pull method), or you can supply no callback, and then SDL will expect you to supply data at regular intervals (push method) with this function.

There are no limits on the amount of data you can queue, short of exhaustion of address space. Queued data will drain to the device as necessary without further intervention from you. If the device needs audio but there is not enough queued, it will play silence to make up the difference. This means you will have skips in your audio playback if you aren\textquotesingle{}t routinely queueing sufficient data.

This function copies the supplied data, so you are safe to free it when the function returns. This function is thread-\/safe, but queueing to the same device from two threads at once does not promise which buffer will be queued first.

You may not queue audio on a device that is using an application-\/supplied callback; doing so returns an error. You have to use the audio callback or queue audio with this function, but not both.

You should not call \mbox{\hyperlink{SDL__audio_8h_a7ffa484f230f6085e271a4f43e678f06}{SDL\+\_\+\+Lock\+Audio()}} on the device before queueing; SDL handles locking internally for this function.

Note that SDL2 does not support planar audio. You will need to resample from planar audio formats into a non-\/planar one (see SDL\+\_\+\+Audio\+Format) before queuing audio.
